Приложение работает следующим образом: при создании нового холста в бд записывается информация о нём и создается директория с индексом в названии, 
сам холст сохраняется в виде фрагментов с высотой не более 5000. Инфоромация о фрагментах (порядок, путь файла и привязка к холсту) вносится в бд.
То есть для холста 20000*50000 получится 10 фрагментов 20000х5000. По памяти хипа джава машина не падает ни при каких значениях высоты. 
При добавлении части изображения высчитываются номера фрагментов, которые нужно изменить, затем извлекаются пути к ним из бд, и идёт перезапись файлов,
остальные фрагменты не изменяются.
Получение участка аналогично, только запись идёт не из пользовательского файла в фрагменты, а наоборот. Если запрошенный участок частично 
выходит за рамки всего холста, то изображение урезается до реальных координат.
Также можно создавать сколько угодно холстов. Чтобы уменишить количество потребляемой памяти можно уменьшить количество ядер (в .yml)
Бд - h2 in memory, то есть сбрасывается после рестарта приложения, сделал так для вашего удобства во время тестов, 
но легко можно перевести на любую бд с хранением информации в файле. Создание таблиц при пуске с помощью миграций.
Путь для хранения файлов указывается при запуске джарника как первый аргумент командой строки, если его нет, то по умолчанию будет сохранять в ресурсы.
(другую директорию по умолчанию можно указать в .yml, первая строка directory)
Юнит тесты не покрывают все возможные кейсы, но проверяют корректность работы для стандартных запросов.
